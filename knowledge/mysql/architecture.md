# MySQL架构与原理

## MySQL体系结构

MySQL采用典型的客户端/服务器架构，主要由以下组件构成：

### 连接池
- 管理客户端连接
- 提供连接复用机制
- 减少连接创建开销

### SQL接口
- 接收SQL语句
- 语法解析和验证
- 返回执行结果

### 分析器
- 词法分析：识别SQL关键字
- 语法分析：生成语法树
- 语义分析：检查权限、表是否存在

### 优化器
- 生成执行计划
- 选择索引使用策略
- 决定JOIN顺序
- 基于成本优化（CBO）

### 执行器
- 调用存储引擎接口
- 执行具体数据操作
- 返回结果集

### 存储引擎
- InnoDB：支持事务、行锁、外键
- MyISAM：不支持事务，表锁，读取快
- Memory：内存存储，临时表

## InnoDB存储引擎

### Buffer Pool
- 缓存数据和索引页
- 默认大小建议为内存的70-80%
- 由多个page组成，默认page size为16KB

### Redo Log
- 记录数据变更操作
- 用于崩溃恢复
- 循环写入，固定大小

### Undo Log
- 记录数据修改前的状态
- 用于事务回滚
- MVCC实现基础

### Doublewrite Buffer
- 防止页断裂
- 先写入双写缓冲区
- 再写入实际数据文件

## MySQL查询执行流程

1. **连接建立** → 验证权限，获取连接
2. **查询缓存**（MySQL 8.0已废弃）
3. **语法解析** → 生成语法树
4. **预处理** → 检查表、列、权限
5. **优化** → 生成执行计划
6. **执行** → 调用存储引擎
7. **返回结果** → 发送给客户端

## MySQL日志系统

### Redo Log（重做日志）
- 物理日志，记录页修改
- 用于崩溃恢复
- innodb_flush_log_at_trx_commit控制刷盘策略

### Undo Log（回滚日志）
- 逻辑日志，记录反向操作
- 用于事务回滚和MVCC

### Binlog（二进制日志）
- 逻辑日志，记录SQL语句
- 用于主从复制
- sync_binlog控制同步策略

## MVCC多版本并发控制

通过Undo Log实现：
- 每个事务看到的数据版本不同
- 读操作不阻塞写操作
- 写操作不阻塞读操作

Read View包含：
- m_ids：活跃事务ID列表
- min_trx_id：最小活跃事务ID
- max_trx_id：下一个事务ID
- creator_trx_id：创建者事务ID
